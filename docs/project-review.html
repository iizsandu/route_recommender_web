Predictive Analysis

Two-tier ML pipeline: a KDE continuous risk surface as the primary model, blended with a LightGBM cell-based classifier as an ensemble second opinion.

Why PR-AUC, not accuracy? Crime is a rare-event problem — most (lat, lng) grid cells have zero crimes in any given week. A model that always predicts "no crime" gets 99%+ accuracy trivially. PR-AUC measures the tradeoff between precision and recall specifically for the positive (crime) class, making it the honest metric here.

Time-based holdout: Train set = data before April 16 2026 (7,862 records). Test set = April 16 – May 16 2026 (935 records). No data leakage — the model never sees future crime locations during training.

Tier 1 — KDE Risk Surface (Primary)

Kernel Density Estimation fits a continuous, smooth probability surface over crime event coordinates. No grid cells needed — any (lat, lng) can be scored instantly.

📐 How KDE Works

Each crime record is a point on the map. KDE places a Gaussian "bump" of probability at each point. The bumps sum to create a continuous surface — denser crime clusters produce higher density values.

Bandwidth = 0.015° (~1.7 km) — tighter than Silverman's rule to capture neighbourhood-level clustering rather than city-wide blur.

Recency decay — each crime contributes weight exp(−age_days / 90), giving 83% of total weight to records ≤ 90 days old.

route_risk = Σwaypoints kde(lat, lng) × female_weight(crime_type) × time_modifier(hour) × dwell_seconds

Naive baseline caveat: The naive model (always predict recent hotspot) scores 0.78 because ~80% of crime is recent and geographically concentrated. Tier 2 — LightGBM Cell Classifier (Ensemble)

A gradient-boosted tree model predicting crime probability per H3 hexagonal grid cell per week. Blended 30% with KDE to improve precision in repeat-crime areas.

🔷 H3 Hexagonal Grid Features

The city is divided into H3 resolution-7 cells (~5 km diameter). For each cell × week combination, three rolling density features are computed:

FeatureDescription
cell_density_30dCrime count in this cell over last 30 days
cell_density_90dCrime count in this cell over last 90 days
neighbour_density_30dCrime count in 6 adjacent H3 cells (spillover signal)
week_of_yearSeasonal pattern signal
cell_id (encoded)Location embedding via label encoder

Label leakage prevention: Rolling features use shift(1) — the model sees last week's density to predict this week's crime. No future data seen.

Low per-category PR-AUC is expected with sparse data (~170–830 events per category). The global model (0.35) is the primary LGB contribution. Per-category models improve recall for specific crime types.

🔀 Ensemble Blending & Risk Banding

Score Formula

final_score = 0.70 × kde_score + 0.30 × lgb_score band = "Low" if score < 0.0713 (city p33) band = "Medium" if score < 0.9142 (city p66) band = "High" if score ≥ 0.9142

Banding thresholds are city-wide percentiles (p33 / p66) calibrated from 400 random Delhi points. Raw scores are never shown to users — only Low / Medium / High bands.

Why 3 Bands, Not Raw Scores?

⚖
Legal — Defamation defence
Specific numbers about specific streets carry higher legal risk in India
🎯
UX — Users need decisions, not decimals
"This route is High risk" is more actionable than "score = 0.847"
🔬
Honest about model uncertainty
KDE on ~4,000 points cannot justify 4-decimal precision

End-to-End Methodology

Complete data flow from Cosmos DB crime records through the ML pipeline into the live application.

Sister repo runs LLM extraction on news articles and writes structured crime records to Cosmos DB. This repo reads those records — no ETL, no duplication.

Model artifacts baked into Docker image at build time — no external storage needed at runtime. Weekly retrain rebuilds and pushes a new image.

Hot reload — backend checks MLflow registry hourly and swaps the in-memory model atomically when a new Production version is promoted.
